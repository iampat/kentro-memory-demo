## 2026-05-02

- 2026-05-02 | research | Verified Python ↔ Witchcraft integration via subprocess on Mac. Wrote `scripts/python_integration_test.py` in the witchcraft sibling repo: 6-doc kentro-shaped corpus, full `readcsv → embed → index → query/hybrid` chain via `subprocess.run`, parser handles Witchcraft's `{score}: {body}\n @ {sub_idx}` block format (body can span multiple lines, blocks separated by `===`). End-to-end: 13ms readcsv + 147ms embed + 18ms index for 6 docs; ~110ms per query (subprocess startup + model load + search). Semantic queries land correct top-k for keyword-aligned questions ("what is the renewal price for Acme" → meeting transcript at 0.86). Two recall caveats observed: T5 weakly embeds bare digit strings (phone-number lookup missed); short docs make the hardcoded 0.7 threshold prune some borderline matches. Both are corpus-quality issues, not integration issues. **Conclusion: Witchcraft + Python via subprocess is a viable v0 integration path.**
- 2026-05-02 | decision | Locked v0 Witchcraft integration approach: **subprocess wrapper around `warp-cli`**. Per-doc `add` via append-to-tenant-TSV + `readcsv → embed → index` (incremental — embed/index only touch pending rows). Per-doc `remove` for v0 via wipe-and-re-ingest (acceptable at the demo's 6-docs-per-tenant scale, ~200ms). PyO3 binding (`kentro-witchcraft` crate via `maturin`) deferred to v0.1 if scale demands it. Rationale: zero new toolchain in the kentro build, works today on every platform Witchcraft builds on, fits comfortably within demo latency budgets. Handoff not changed — Witchcraft commitment unaffected; this is implementation detail.
- 2026-05-02 | research | Validated Dropbox Witchcraft (`github.com/dropbox/witchcraft` @ master) end-to-end on Mac arm64 (M2 Max, Metal). Build chain ran cleanly via `make warp-cli` after installing rustup (rustc 1.95.0). Footprint: HF download 422 MB (`xtr-base-en/`) + 210 MB (`xtr.safetensors`) + 62 MB (`assets/` GGUF + tokenizer/config) + 1.3 GB (`target/` build dir) + 828 MB (Python venv). Persistent runtime: 62 MB assets + binary. Indexed nfcorpus (3633 medical-research docs): `readcsv` 0.7s, `embed` 1m25s, `index` 10.7s, `mydb.sqlite` ends at 181 MB. Semantic `query` and `hybrid` both return correct top-k for medical queries (e.g. "milk intake / acne" returns the milk-acne association abstracts at scores 0.92/0.91/0.91). Subprocess wall-time per query: ~140ms steady-state (~270ms cold). README's 21ms p95 is the in-process latency without subprocess startup. CLI surface confirmed: `clear | readcsv | embed | index | reindex | query | hybrid | querycsv | hybridcsv | fulltextcsv | score`. Output format is plain text `{score}: {body} @ {sub_idx}` with `===` separators — no JSON flag. DB path hardcoded to `./mydb.sqlite`, assets to `./assets/`.
- 2026-05-02 | research | Mapped Witchcraft Rust library API (`src/lib.rs` `pub use`s + `src/db.rs` `impl DB`). `DB` exposes `add_doc`, `add_docs_batch`, `remove_doc(uuid)`, `delete_with_filter`, `clear`, `begin/commit/rollback_transaction`.
